# 체크포인트 메시지는 신뢰도 prefetch로 생성한다

체크포인트의 *체감 latency*가 0에 수렴해야 한다는 제품 요구가 있다. 그런데 LLM 메시지 생성은 가장 빠른 클라우드도 800ms~2.5s가 걸려 발동 시점에 호출하면 명백히 늦다.

본 제품의 발동 룰은 *신뢰도 × 체류*이므로 임계 통과 *훨씬 전부터* "곧 발동 가능성"이 누적된다. 이 곡선 자체를 compute budget으로 사용한다 — **예비 임계(신뢰도 ~0.4)를 넘는 순간 백그라운드에서 LLM 호출을 시작**, 최종 임계(신뢰도 ~0.7 × dwell 90s) 통과 시점엔 캐시된 응답을 즉시 표시한다. 사용자의 *실시간 변수*(누적 분, 자세 각도, 세션 N번째 연장)는 LLM 응답의 *슬롯*으로만 두고 표시 직전에 바인딩한다.

사용자가 알아서 멈춰 신뢰도가 떨어지면 캐시는 폐기한다(stale data 방지).

## Considered Options

- **on-demand 호출** — 가장 단순하지만 latency 요구를 정면으로 위반.
- **온디바이스 LLM** — latency를 0에 가깝게 만들 수 있지만 모델 크기·배터리 부담이 크고 메시지 품질도 떨어짐. 추후 비용/프라이버시 재검토 시 다시 후보가 될 수 있음.
- **순수 템플릿** — latency 0이지만 "매번 다른 톤"이라는 제품 가치를 잃음.

## Consequences

- LLM 인프라는 일단 **클라우드 호출**을 전제로 한다. 단 *프라이버시·비용·오프라인 동작* 결정은 **별도 ADR 보류** — 클라우드 vs 온디바이스 LLM 선택은 본 ADR과 직교한다(둘 다 prefetch 패턴을 따른다).
- 신뢰도가 예비 임계에 진입했다가 빠지기를 반복하면 *낭비 호출*이 발생할 수 있음 → 예비 임계 진입 후 짧은 hold-off 또는 *호출 디바운스*가 필요.
- 메시지 슬롯 변수(누적 카운터, 현재 자세 등)는 LLM 응답 *외부*에서 관리. LLM은 *톤/문장 변주*만 책임, 사실 수치는 우리 시스템이 책임 — 환각 차단.
