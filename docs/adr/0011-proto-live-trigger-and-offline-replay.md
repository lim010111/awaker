# 프로토타입/베타의 라이브 트리거는 AS teacher 룰 하나, 권한-경량 판별은 replay로 오프라인 평가한다

v1 프로토타입 범위를 *E2E tracer bullet + 베타 데이터 수집*의 결합 빌드로 정했다 (센서 파이프라인 → 판별 → 오버레이 체크포인트 → 이탈 측정, 동시에 IMU raw + AS 스크롤 운동학 로깅. LLM은 [[local-template-pool]] stub). 그러면 student 모델이 아직 없는 이 단계에서 *무엇이 체크포인트를 발동시키는가*가 결정돼야 한다.

**라이브 트리거는 AS 운동학 teacher 룰([[0010-detection-label-kinematic-proxy]]) 하나만 쓴다.** 수집 목적으로 어차피 빌드에 있는 신호를 트리거로 재사용 — 스크롤을 직접 보므로 판정이 정확해 *체크포인트 UX 가설 검증이 판별 오탐에 오염되지 않는다*. 자이로 휴리스틱·student 후보는 폰에서 돌리지 않고, **로깅된 raw 스트림을 오프라인 replay**로 흘려보내 "그때 발동했을까"를 채점한다 — 같은 로그에 후보를 무한 재실험 가능, 라이브 대비 정보 손실 0.

이 결정의 성립 조건이자 프로토타입의 숨은 핵심 산출물: **로깅 스키마의 acceptance criteria = "로그만으로 디바이스 없이 판별을 오프라인 재현 가능"**. 센서 raw 전체(자이로/가속도 50–100Hz, 조도, 화면 상태) + AS 스크롤 이벤트 + 포그라운드 이벤트가 공통 타임라인에 정렬되어야 하며, 윈도우 분절은 X만으로 계산 가능해야 한다(ADR-0010 누수 관리 (a)와 정합).

## Considered Options

- **자이로 휴리스틱 라이브 트리거** — 출시와 같은 신호 경로를 일찍 실전 테스트하지만, 데이터 0 상태의 수제 휴리스틱이라 오탐 시 "판별이 틀렸나 UX가 약한가"를 분리 못 함. UX 검증과 판별 검증이 서로 오염. 비채택.
- **이중 라이브 트리거 (AS 발동 + 자이로 그림자 비교)** — replay가 동일 정보를 더 싸게 주므로 라이브로 돌릴 이유가 구현비뿐. 비채택.
- **수동/시간 기반 트리거** — 판별과 UX의 결합을 아예 검증 못 함. 비채택.

## Consequences

- teacher 룰의 *임계/수식 1차 튜닝*이 프로토타입 산출물로 앞당겨짐 — ADR-0010 "남은 미결" 중 GT 룰 수식이 여기서 초안 확정.
- replay 파이프라인이 사실상 *모델 개발 인프라의 시작* — 이후 student 학습/백테스트가 같은 로그·같은 하네스를 씀.
- 프로토타입이 검증 **못 하는** 것 명시: 출시 형태(자이로 전용)의 실시간 판별력. 이는 replay 추정으로만 접근하고, 실기기 검증은 student 모델 탑재 빌드에서.
- ADR-0001의 신뢰도 곡선 prefetch는 이 단계에서 미검증(LLM stub) — 룰 기반 근사 신뢰도로 뼈대만 유지.
- 출시 빌드에서 AS 트리거 경로 제거는 ADR-0004 약속 그대로 유효.
