# Progressive Onboarding과 수집 정보 거버넌스

행동 변화 제품의 가장 큰 적은 *첫 30초 이탈*. 깊은 온보딩(나이/직업/만성질환/Health Connect)을 첫 실행에 몰아넣으면 *우리 핵심 가치를 체험하기도 전에* 사용자가 떨어진다. 동시에 메시지 개인화는 *코어가 아니라 보너스* — 본 제품은 나이/건강 정보 없이도 동작한다.

따라서 온보딩은 **3단계 점진적**으로 운영한다.

## 단계

**0일차 — 미니멈 (~30초)**
- ① 1차 목표 선택: `집중력` / `수면` / `일반`
- ② 필수 권한 부여: UsageStatsManager, 알림, SYSTEM_ALERT_WINDOW
- 끝. 즉시 핵심 가치 체험 가능.

**3~7일차 — 보통 (사용자가 *최소 1번* 체크포인트 경험 후)**
- 앱 안 카드 형태로 부드럽게: 나이대 (`학생` / `20대` / `30대+` / `시니어`), 직업 카테고리 (`학생` / `직장인` / `기타`), 주요 시청 시간대.
- *각 항목 거부 가능*. *왜 묻는지* 1줄 설명 필수 ("이 톤으로 메시지를 보내드리려고요").

**2주차+ — 깊이 (지속 사용자 한정, 완전 선택)**
- Health Connect 연동, 만성질환 카테고리, 수면 목표.
- Health Connect는 *읽기 1회 → 카테고리 라벨만 보관*. 지속 모니터링은 v2+ 검토.

## 수집 정보 거버넌스

- **저장 위치**: 기본 *온디바이스*. 서버는 *명시 옵트인* 시 백업 용도로만 — 암호화 필수, 사용자가 언제든 삭제 가능.
- **LLM 프롬프트로의 노출**: [[bucketing]] 규칙(ADR-0005) 그대로 적용 — 나이 raw 대신 `"학생"` 라벨, 만성질환 raw 대신 `"건강 민감 시간대"` 같은 익명 토큰.
- **삭제 권리**: 사용자는 *언제든* 수집 정보 삭제 가능. 삭제 시 LLM 프롬프트는 *기본 일반 톤*으로 자동 회귀.
- ***early-user state* 처리**: 메시지 생성기는 *비어있는 필드를 자연스럽게 무시*해야 함. 0일차 사용자 = "모든 personalization 필드 null" → LLM 프롬프트에 단순히 그 슬롯이 빠짐.

## Considered Options

- **풍부한 1회성 온보딩 (O-C)**: 깊은 개인화 가능하나 *설치 이탈률 ↑↑*. 핵심 가설 검증 비용에 비해 손실 큼.
- **온보딩 없음 (모두 익명)**: 가장 가벼우나 메시지 톤이 *완전 일반*. 너의 "그룹별 적응형" 비전 포기.
- **서버 우선 저장**: 다중 기기/백업 편의 ↑지만 본 제품 신뢰 약속과 충돌. 비채택.

## Consequences

- 메시지 생성 로직은 *부분 정보 입력*을 처음부터 일급으로 다뤄야 함 — 단계별 사용자 모두 자연스러운 톤이 나와야.
- 3~7일차/2주차+ 카드를 *언제·어떤 컨디션*에 띄울지는 별도 운영 결정 (예: "체크포인트 N회 경험 후 1회만").
- "백업 옵트인"이 *유료 기능*이나 *프리미엄* 전환 후보 — 본 ADR에선 비결정.
- Health Connect 연동은 *추가 권한 + 정책 검토* 필요. 2주차+에 도달한 사용자 비율이 충분히 클 때 구현 우선순위.
- 깊은 개인화는 *데이터 보고 v2 결정*. v1 핵심 가설은 *온보딩 없어도 통하는지* 부터 검증.
