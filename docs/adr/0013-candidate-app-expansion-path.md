# 후보 앱 확장: OS가 역할을 정의하는 브라우저만 자동 탐지, 나머지는 기본 셋 → 사용자 선택 UI 경로

브라우저에서도 무지성 스크롤을 잡기로 했다 (이슈 08). 브라우저는 종류가 많고
사용자마다 쓰는 것이 달라, 후보 셋을 어떻게 채우는가가 결정 대상이 됐다.
CONTEXT.md의 [[후보 앱]] 정의("기본 셋 + 사용자가 직접 추가/제외 가능")는 이미
확장을 상정하고 있으므로, 이 ADR은 그 *채움 메커니즘*의 결정이다.

**브라우저는 OS에 묻는다.** Android는 브라우저라는 역할을 시스템 차원에서
정의한다 — 범용(호스트 무제한) `https` VIEW 인텐트 핸들러. 설치된 브라우저를
런타임에 질의해 후보 셋에 자동 포함한다(서비스 시작 시 1회 해석). 프로브를
범용 호스트로 잡아 도메인 특정 App Link 앱(YouTube가 youtube.com을 처리하는
식)은 딸려오지 않는다. 이 방식이 성립하는 건 **OS가 역할의 정의를 제공하는
경우뿐**이다 — "영상 플랫폼"이나 "피드 앱"에는 그런 정의가 없으므로 같은
자동 탐지를 일반화할 수 없다.

**자동 포함은 기본값일 뿐, 사용자 제외가 항상 우선한다.** 제품 원칙(체감
변수는 기본값만 제공하고 사용자 조정 허용)의 적용이다. 프로토타입엔 조정 UI가
없어 실효는 베타부터지만, 우선순위 자체는 지금 확정한다 — "Chrome은 업무용이라
빼고 싶다"는 자동 포함으로 막을 수 없어야 한다.

**그 외 앱의 확장 경로: 기본 셋 하드코딩(프로토타입) → 사용자 선택 UI(베타).**
"내가 무지성으로 스크롤하는 앱"은 본질적으로 개인적이라, 자동 추정이 아니라
사용자 선택이 정답이다. 프로토타입은 알려진 피드 앱 하드코딩으로 충분하다.

**teacher 룰 임계는 단일 Config를 유지한다 — 단, 앱 부류별 분화가 맞는 방향
이라는 인식 아래의 보류다.** 브라우저를 YouTube처럼 쓰지 않고 YouTube를
브라우저처럼 쓰지 않는다 — 앱 형태에 따라 스크롤 운동학 자체가 다르므로,
장기적으로는 부류별 임계가 자연스럽다. 지금 나누지 않는 이유는 하나: 부류별
임계값을 정할 실증 데이터가 없다. 데이터 없이 쪼개면 추측 임계 두 벌이 될
뿐이다. **재검토 트리거**: 이슈 08 AC의 브라우저 정독 오탐 체감 기록과
self-annotation([[0010-detection-label-kinematic-proxy]])에서 브라우저 세션
오탐이 실증될 때.

## Considered Options

- **브라우저 하드코딩 목록** — Chrome·Samsung Internet 등 상위 N개 나열.
  사용자별 다양성을 흡수 못 하고 목록 관리 비용만 남는다. OS 정의가 있는데
  안 쓸 이유가 없다. 비채택.
- **카테고리 메타데이터(appCategory) 자동 분류** — 앱의 VIDEO/SOCIAL 자가
  신고를 이용한 후보 자동 확장. 신고 안 한 앱이 많아 커버리지가 구멍투성이고,
  설치 앱 전체 열람(QUERY_ALL_PACKAGES)이 필요해 스토어 정책 리스크. 자동
  확정용 비채택 — 베타 사용자 선택 UI의 *추천 프리필* 후보로만 남긴다.
- **앱 부류별 teacher 임계 즉시 분화** — 방향은 맞다고 보나 임계를 정할
  데이터가 없어 보류 (본문 재검토 트리거 참조). 비채택이 아니라 연기.
- **브라우저는 룰 대상에서 제외하고 로깅만** — 커뮤니티·뉴스 무한 스크롤 등
  브라우저 무지성 스크롤이 실제 타깃이므로 반쪽 확장이 된다. 비채택.

## Consequences

- 후보 셋이 기기마다 다르다 — 로그 분석·replay 시 세션 pkg의 다양성을 전제로
  해야 한다.
- 브라우저 정독(기사·문서 읽기)도 스크롤을 동반하므로 오탐 리스크가 피드 앱
  보다 높다 — [[0007-v1-north-star-and-guardrails]] 오탐 신고율 가드레일(<10%)
  압박 가능. 위 재검토 트리거가 이를 감시한다.
- 서비스 기동 후 새로 설치된 브라우저는 서비스 재시작 시 반영된다 (해석이
  시작 시 1회 — 프로토타입 허용).
- 임계 분화가 발동되면 `TeacherRule.Config`를 부류별로 나누는 진화가 된다 —
  replay 하네스가 같은 분기를 미러링해야 하므로 부류 판별 기준(패키지 → 부류
  매핑)이 로그만으로 재현 가능해야 한다 ([[0011-proto-live-trigger-and-offline-replay]]
  의 오프라인 재현 AC와 정합).
