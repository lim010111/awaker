# 판별 엔진은 권한-경량 센서 융합으로 입력을 구성한다

Android에서 *다른 앱의 터치/스크롤 이벤트*에 직접 접근하려면 AccessibilityService가 필요한데, 이 권한은 Play Store 정책상 *접근성 보조*가 진짜 목적이 아니면 통과가 어렵고 사용자 첫인상 거부율이 매우 높다 — 권한 거부 시 앱 무용지물 구조는 잔존률을 깬다. 우리는 **AccessibilityService 없이** [[detection-engine]] 의 입력을 구성한다:

- **자이로 + 가속도** (3축, 50–100Hz, 무권한)
- **조도 센서** (시나리오 B의 "어두운 방" 신호, 무권한)
- **화면 상태 / 배터리 / 시간대** (무권한 ~ 가벼운 권한)
- **UsageStatsManager** — 후보 앱 포그라운드 시작/종료 시각만 (PACKAGE_USAGE_STATS, 민감하지만 정당 사용 통과 사례 많음)

스크롤 이벤트는 *직접 신호*로 얻지 않고 *자이로 미세 흔들림으로 간접 추론*한다. 이 결정의 핵심 직관: 스크롤 신호를 "컨닝페이퍼"로 받지 않고 센서 융합만으로 무지성 스크롤을 추론하는 능력 자체가 우리의 **진짜 AI 자산** 이다. 통하면 추적 어려운 해자가 됨.

## Considered Options

- **자이로/가속도만 (S1)** — 권한 0이지만 *정지 상태에서 영상만 보는* 케이스를 거의 못 잡고 정확도 천장이 낮음.
- **자이로 + AccessibilityService (S2)** — 모델 정확도는 최고지만 *정치적 위험*(Play Store 정책 강화 흐름)과 *사용자 거부율*이 v1에 치명적. **v2 옵션 권한**으로 재검토.

## Consequences

- ML 문제가 *더 어려워짐* — 자체 수집 데이터 규모와 모델 품질이 가설 검증의 단일 병목.
- **베타/자체 수집 단계에서만 AccessibilityService를 *지도학습 라벨* 용도로 한시 사용 OK** — 베타 참가자에게 명시 동의 받고 *목적·기간·삭제 시점*을 고지. 출시 빌드엔 AS 코드 경로 제거.
- 신뢰도 임계는 *오탐 ↓ 양성 누락 ↑* 방향으로 보수적 튜닝. 사용자 입장에서 "안 잡혔다"는 *부정적 마찰이 없는* 실패라 잔존률 보존.
- **체크포인트 표시**만 SYSTEM_ALERT_WINDOW가 필요 — AccessibilityService 대비 정치적 부담 *훨씬 가벼움*.
- *v1에서 통하면* 그 자체로 시장 차별점("스크롤 이벤트도 안 보면서 잡아낸다"). 통하지 않으면 S2를 옵션 권한으로 추가.
